# Domain Pitfalls: On-Device Scam Text Classifier — Adding to Existing System

**Domain:** On-device NLP classification (scam detection) with synthetic data, knowledge distillation, Int8 QAT, TFLite mobile deployment
**Researched:** 2026-04-01
**Scope:** Pitfalls specific to ADDING these capabilities to an existing React Native + TFLite pipeline. Generic ML advice excluded.

---

## Area 1: Synthetic LLM-Generated Training Data

### Critical Pitfall 1.1 — Evaluation Set Contamination (The Accuracy Illusion)

**What goes wrong:** You generate synthetic scam examples with an LLM (e.g., Gemini), split them into train/test, train the classifier, and get 95%+ accuracy. You ship it. In production, the model fails on real scam texts. The test set was generated by the same LLM with the same prompts, so "high accuracy" measured how well the model learned the LLM's output style — not whether it detects real scams.

**Why it matters here:** The current MobileBERT failure is already a generalization problem (overfitting to SMS spam distribution). If the evaluation methodology does not change when training data generation changes, you get a new model with the same failure mode and a false sense of confidence. Research at IEEE Xplore (2025) specifically on LLM-generated scam datasets confirms models achieve "high accuracy on generated test sets" but warns this does not transfer to real-world conversations.

**Prevention:**
- Hold out a real-world evaluation set before generating any synthetic data. Collect 200–500 real scam examples from public sources (FTC complaint data, r/scams, Snopes, PhishTank for phishing text) BEFORE running any generation.
- Evaluate the final model exclusively on this real-world holdout. Never include LLM-generated examples in the primary evaluation metric.
- If no real-world data is available, at minimum test against the existing SMS spam datasets the model already fails on — real diversity, even from a narrow domain, will surface generalization failures the synthetic test set will not.

**Phase relevance:** TEXT-01 (dataset generation) and TEXT-02 (architecture benchmark) — evaluation methodology must be locked before any model is trained.

### Critical Pitfall 1.2 — Stylistic Mode Collapse Across Scam Vectors

**What goes wrong:** You prompt the LLM to generate examples across 6–8 scam types (crypto, romance, tech support, etc.). The LLM generates syntactically diverse text but applies the same rhetorical template across all types: urgent tone, authority appeal, link. The classifier learns "urgency + authority + link = scam" rather than the vector-specific linguistic fingerprints. It then fails on low-urgency scams (slow-burn romance scams, long-con investment narratives) which are the hardest and most costly to miss.

**Why it matters here:** The token distribution of synthetic data from a single LLM is relatively uniform compared to real user-generated text. Research from ArXiv (2503.14023) confirms that "training only on synthetic data performs notably worse on downstream tasks due to the uniform token distribution." For a security classifier, the samples you most need coverage on (novel, patient scam patterns) are exactly what a single-LLM generation pipeline will underrepresent.

**Prevention:**
- Use at least two different LLMs for data generation (e.g., Gemini 2.5 Flash + an open-source model like Llama 3) and compare the vocabulary diversity between their outputs before training. If Jensen-Shannon divergence on token unigrams is very low, you have mode collapse.
- Explicitly generate "hard negative" examples: texts that use scam rhetoric but describe legitimate things (urgency in a sale, authority in a legal notice, links in a newsletter). Without these, the model will flag benign messages as scams.
- Add structural variety prompts: "Write this as a text message," "Write this as a WhatsApp message with typos," "Write this in the style of a non-native English speaker."

**Phase relevance:** TEXT-01 — data generation phase. The multi-agent or multi-LLM generation strategy should be specified before generation begins, not retrofitted after you see poor precision.

### Moderate Pitfall 1.3 — Prompt Leakage Into the Classifier's Decision Boundary

**What goes wrong:** Your generation prompts include the label explicitly ("Generate a scam message about cryptocurrency"). The generated examples are so topically focused that the classifier learns topic (cryptocurrency, IRS, lottery) as the scam signal instead of the communicative intent. The model flags any mention of cryptocurrency as a scam, including legitimate crypto exchange notifications.

**Prevention:**
- Review generated examples for topical over-specificity. If all crypto scam examples contain "Bitcoin wallet" and all legitimate examples don't, you have a topic-biased dataset, not an intent-biased one.
- Generate legitimate examples within the same topic domains as your scam examples: legitimate bank notifications to contrast with bank impersonation scams, legitimate prize announcements to contrast with lottery scams.
- Check final dataset: for each scam topic vector, there should be at least as many legitimate examples within that same topic domain.

**Phase relevance:** TEXT-01.

---

### Moderate Pitfall 1.4 — LLM Safety Filtering Removes the Hardest Examples

**What goes wrong:** Production LLM APIs (including Gemini) will refuse or soften the most realistic, high-harm scam text generation requests (detailed impersonation of government agencies, sexual grooming patterns, realistic financial fraud scripts). The generated dataset skews toward mild, obviously-fake scam language that the model handles easily. The examples that are hardest to detect in practice are absent from training.

**Prevention:**
- Use a local model (Llama 3, Mistral, or a fine-tuned model without safety filters) for the highest-risk categories, or use indirect prompting ("Write a message that a malicious actor might send, framed as educational awareness material").
- After generation, manually review the distribution of realism. If all examples read as obvious scams, the training data is too easy.
- Consider using real historical scam text (with any PII stripped) as seed examples to anchor the LLM generation toward authentic linguistic patterns.

**Phase relevance:** TEXT-01.

## Area 2: Knowledge Distillation

### Critical Pitfall 2.1 — Teacher Model Inherits the Generalization Problem

**What goes wrong:** You fine-tune RoBERTa or DeBERTa on the same synthetically generated scam dataset and use it as the teacher. The teacher achieves high accuracy on the synthetic test set (see Pitfall 1.1). You then distill into MobileBERT or DistilBERT. The student faithfully learns the teacher's decision boundary — which is the wrong decision boundary. You now have a compressed model that compresses the wrong behavior efficiently.

**Why it matters here:** This is the highest-risk compounding failure in this project. The current model is broken due to bad training data. If the teacher is trained on bad synthetic data, distillation propagates and concentrates that failure. You cannot use the teacher's soft label accuracy as validation — you need the independent real-world holdout from Pitfall 1.1 to validate the teacher before distillation begins.

**Prevention:**
- Gate the distillation phase on a teacher accuracy threshold against the real-world holdout set (not the synthetic test set). Define this threshold before training begins: e.g., teacher must achieve >0.80 F1 on real-world holdout before distillation proceeds.
- Treat teacher selection as a separate phase milestone with its own validation criteria.

**Phase relevance:** TEXT-04 (knowledge distillation). This pitfall justifies making teacher validation a hard gate before TEXT-04 begins.

---

### Critical Pitfall 2.2 — Over-Confident Teacher Soft Labels at Low Temperature

**What goes wrong:** The fine-tuned teacher produces near-one-hot outputs on the training examples it has seen. At temperature T=1, soft labels carry almost no inter-class information ("dark knowledge"). You are effectively training the student on hard labels despite using distillation, and you lose the primary benefit of the technique.

**Why it matters here:** BERT-family models fine-tuned on binary classification tasks (scam/not-scam) push logits to extreme values quickly. The soft labels become [0.001, 0.999] instead of [0.15, 0.85], and the temperature scaling has to be aggressive (T=3–5) to recover meaningful signal. If temperature is not tuned, the student training is equivalent to cross-entropy on hard labels — which you could do without a teacher.

**Prevention:**
- Calibrate the teacher model post-fine-tuning using temperature scaling on a held-out calibration set before generating soft labels. Measure ECE (Expected Calibration Error) before and after.
- Start distillation experiments at T=4 for a binary scam classifier (higher than the standard T=2 recommendation), and sweep T={2, 3, 4, 5} on a small student, evaluating on the real-world holdout, not the distillation training loss.
- Use a mixed loss: KL divergence on soft labels (temperature-scaled) + cross-entropy on hard labels. The ACL 2022 paper on distillation for QAT of transformers shows using MSE on attention scores alone is insufficient; output-level soft label calibration is required.

**Phase relevance:** TEXT-04.

---

### Moderate Pitfall 2.3 — Architecture Mismatch Between Teacher and Student Breaks Intermediate Layer Distillation

**What goes wrong:** You attempt to distill not just the final output logits but also intermediate attention maps (as done in TinyBERT). If the teacher (DeBERTa) and student (MobileBERT or DistilBERT) have different numbers of attention heads or different hidden dimensions, the layer-to-layer mapping requires a projection layer. Teams frequently skip this projection or map to the wrong layers and see the distillation training converge to a worse result than simple logit-level distillation.

**Prevention:**
- For this project, start with output-level distillation only (soft labels from the teacher's final classification head). Only add intermediate-layer distillation (attention maps, hidden states) if output-level distillation fails to meet accuracy targets.
- If intermediate distillation is needed, explicitly define the layer mapping table before writing any training code: teacher layer N → student layer M via a learnable linear projection.

**Phase relevance:** TEXT-04.

---

### Minor Pitfall 2.4 — Distillation on a Frozen Student Tokenizer Breaks With Architecture Swap

**What goes wrong:** You swap the base architecture from MobileBERT to DistilBERT during the architecture benchmark (TEXT-02), but continue using the existing `vocab.txt` from the broken MobileBERT pipeline. DistilBERT uses the same WordPiece vocabulary as BERT-base, so this appears to work. But if you switch to a non-BERT tokenizer (e.g., RoBERTa's BPE), the `TextTokenizer.ts` in the existing app pipeline will silently produce wrong token IDs against the new model's embedding matrix.

**Why it matters here:** The existing `TextTokenizer.ts` loads `vocab.txt` from assets and implements WordPiece manually. It is tightly coupled to the BERT vocabulary. This is exactly the input format issue that broke the current model. Changing architecture without updating and validating the tokenizer-model pairing re-introduces the same bug class.

**Prevention:**
- Lock the tokenizer-architecture pairing as a hard constraint at the start of TEXT-02. The evaluation must test the full pipeline: tokenizer → model → output, not the model in isolation with a Python tokenizer.
- Add an integration test in Python that feeds identical text through (a) the HuggingFace Python tokenizer for the chosen architecture and (b) a port of the `TextTokenizer.ts` logic, and asserts the token ID arrays are identical before any TFLite conversion.

**Phase relevance:** TEXT-02 and TEXT-06.

## Area 3: Int8 Quantization-Aware Training (QAT)

### Critical Pitfall 3.1 — TFLite Int8 Quantization Silently Falls Back to Float32 Inputs

**What goes wrong:** You apply Int8 QAT and run `TFLiteConverter` with `optimizations = [DEFAULT]` and `target_spec.supported_types = [INT8]`. The converted model appears quantized. But TFLite's full-integer quantization in TF2 allows float input/output types by default — the input tensor still accepts `float32`, the quantization happens internally, and `input_details[0]['dtype']` reports `numpy.float32`. The production code in `TextClassifierService.ts` passes tokens as a buffer typed for int32 (BERT token IDs). The runtime does not throw; it silently produces garbage outputs.

**Why it matters here:** This exact pattern — a model that appears to work without errors but produces meaningless outputs — is already the failure mode of the current broken model. This pitfall would reproduce that failure with the new model. The GitHub issue #40681 on tensorflow/tensorflow documents this silent failure explicitly.

**Prevention:**
- After conversion, assert `input_details[0]['dtype'] == numpy.int32` for BERT token ID inputs and `output_details[0]['dtype'] == numpy.float32` (or int8 with known zero_point/scale) in the conversion validation script.
- Use `converter.inference_input_type = tf.int32` explicitly for token ID inputs (not float). Token IDs are integers — they should never be cast to float32 before passing to the model.
- Add a round-trip test in the research notebook: feed 5 known texts through the Python tokenizer → Python TFLite interpreter → assert outputs match the pre-conversion model outputs within a defined tolerance (e.g., <0.05 absolute difference on the scam probability score).

**Phase relevance:** TEXT-05 (QAT) and TEXT-06 (deployment). The round-trip test must exist before the model is copied to `canaryapp/assets/models/`.

---

### Critical Pitfall 3.2 — BERT Softmax and LayerNorm Quantization Instability

**What goes wrong:** BERT's attention softmax is unbounded and changes rapidly. BERT also has documented "outlier dimensions" in its hidden states that have activation values orders of magnitude larger than the median. Int8 quantization maps the full activation range to 256 values. If the outlier activations dominate the scale calibration, all other activations collapse into a few quantization buckets, destroying the model's representational capacity for normal cases.

**Why it matters here:** This is not a theoretical concern — the I-BERT paper (Kim et al., 2021) specifically documents that softmax, GELU, and LayerNorm must be handled at INT32 precision in integer-only quantization, and that naive Int8 quantization of these layers causes measurable accuracy degradation. For a security classifier where precision >0.90 is required (per PROJECT.md's false-positive constraints), a 2–5% accuracy drop from this cause may violate the product requirement.

**Prevention:**
- Use QAT rather than post-training quantization (PTQ) specifically because QAT allows the model to learn clipping thresholds during training, partially mitigating the outlier problem.
- Configure `tfmot.quantization.keras.quantize_model` with the default `LastValueQuantizer` for all layers, then verify the quantization ranges learned during QAT by inspecting `quantize_annotate_layer` output. If the max activation range for any layer is >100x the median range, that layer has outlier activations and may need per-channel quantization or INT16 fallback.
- Accept that a small accuracy loss (0.5–1.5%) is expected and acceptable for Int8; define the acceptable degradation threshold in the notebook before QAT, so the result is evaluated against a known bar rather than post-hoc.

**Phase relevance:** TEXT-05.

---

### Moderate Pitfall 3.3 — QAT Requires Longer Training Than Expected for BERT Fine-Tuning

**What goes wrong:** You apply `quantize_model()` to a fine-tuned BERT and continue fine-tuning for the same number of epochs as the original fine-tuning. The QAT model converges to a worse result than PTQ. The underlying cause is that the quantization-aware fake-quantization nodes introduce gradient noise early in training, requiring more epochs at a lower learning rate to stabilize.

**Prevention:**
- Start QAT from a fully fine-tuned (not randomly initialized) checkpoint. Never apply QAT to a model being trained from scratch on the task.
- Use a learning rate 10x lower than the fine-tuning learning rate for QAT (typically 1e-5 for fine-tuning → 1e-6 for QAT).
- Train for at least 2x the epochs of the original fine-tuning run under QAT. Budget for this in the research timeline.

**Phase relevance:** TEXT-05.

### Minor Pitfall 3.4 — SELECT_TF_OPS Dependency Makes the Model Incompatible with react-native-fast-tflite

**What goes wrong:** Some BERT-family models (particularly RoBERTa or DeBERTa converted from PyTorch/HuggingFace) require `SELECT_TF_OPS` during TFLite conversion because they use TensorFlow ops not in the built-in TFLite op set. Converting with this flag produces a model that requires the `tensorflow` dependency in the TFLite runtime, not just `tensorflow-lite`. `react-native-fast-tflite` bundles a standard TFLite runtime and does not include the `SELECT_TF_OPS` extension by default. The model file loads, but inference throws an "unsupported op" error at runtime.

**Prevention:**
- Test the TFLite model with the standard TFLite Python interpreter (not the TensorFlow Python interpreter) before considering it production-ready. If it requires SELECT_TF_OPS, the model cannot be deployed as-is.
- Prefer architectures with full TFLite built-in op support: MobileBERT and DistilBERT convert cleanly with built-in ops. DeBERTa v2+ and RoBERTa-large may require SELECT_TF_OPS.
- If a teacher model requires SELECT_TF_OPS but the student does not, that is acceptable — the teacher runs only in the Python research environment, never on-device.

**Phase relevance:** TEXT-02 (architecture benchmark must test TFLite compatibility), TEXT-06.

## Area 4: Integrating the Retrained Model into the Existing App Pipeline

### Critical Pitfall 4.1 — New Model Silently Accepted With Wrong Input Shape

**What goes wrong:** The existing `TextClassifierService.ts` constructs an `Int32Array` of shape `[1, 128]` (batch=1, sequence_length=128) and passes it to `react-native-fast-tflite`. The new quantized model is exported with sequence_length=64 (a common optimization). The JSI zero-copy buffer is the wrong size. `react-native-fast-tflite` does not validate input shape against the model's expected shape before running inference — it passes the buffer to the TFLite interpreter, which may silently truncate, pad, or error at the native layer without a JavaScript exception being thrown.

**Why it matters here:** `CONCERNS.md` already documents that the existing model fails silently — `classify()` throws instead of gracefully degrading. Adding a shape mismatch to a pipeline with no error handling and 207 `console.log` calls makes this failure mode nearly invisible in production.

**Prevention:**
- Before swapping the model file, read `model.inputs` metadata in the research environment and assert the sequence length matches what `TextClassifierService.ts` constructs. If you change sequence length, change the constant in the TypeScript code and add a comment referencing the model version.
- Add a startup validation in `TextClassifierService.ts`: after loading the model, assert `model.inputs[0].shape[1] === EXPECTED_SEQUENCE_LENGTH`. Throw a descriptive error if mismatched, do not swallow it.
- Define the sequence length as a named constant shared between the Python conversion script and `TextClassifierService.ts` (e.g., emit it into a JSON metadata file alongside the `.tflite` file and read it at runtime).

**Phase relevance:** TEXT-06 (model deployment to canaryapp).

---

### Critical Pitfall 4.2 — Output Tensor Index Change Produces Inverted or Wrong Confidence Scores

**What goes wrong:** The existing model outputs a single tensor at index 0 with shape `[1, num_classes]`. The new model (especially if distilled with additional output heads or if the architecture changes) may output multiple tensors, or may output logits at index 0 and softmax probabilities at index 1. The existing pipeline reads `outputData[0]` as the scam confidence score. If the new model puts logits at index 0 (range: -∞ to +∞) instead of probabilities (range: 0–1), the existing threshold logic (`> 0.7 is danger`) breaks silently — every input either fires as a false positive or never fires at all.

**Prevention:**
- In the research environment, after conversion, print `interpreter.get_output_details()` and assert the output tensor index, shape, and dtype match what `TextClassifierService.ts` expects. Document this in the conversion script as a required assertion.
- Standardize the output contract: the `.tflite` model must output a single tensor at index 0 with shape `[1, 2]` (two-class softmax probabilities), and the scam class index must be consistent. Add a comment in the conversion script specifying which class index corresponds to "scam."
- After deploying to the app, test with known-scam and known-safe text before removing the `console.log` statements that print raw scores — they are currently noise but serve as the only observable signal during integration testing.

**Phase relevance:** TEXT-06.

---

### Critical Pitfall 4.3 — Model File Swap Without Removing Old File Causes Stale Model Loading

**What goes wrong:** `CLAUDE.md` explicitly warns: "Do not add files to `canaryapp/assets/models/` without removing old versions." `ModelLoaderService.ts` uses a hardcoded filename `mobilebert_scam_intent.tflite`. If the new model is given a different filename (e.g., `distilbert_scam_v2_int8.tflite`) without updating the reference in `ModelLoaderService.ts`, the service loads the old broken model silently. If the new model is given the same filename but the old file is not removed from git, Metro bundler may cache the old asset.

**Prevention:**
- Establish a model versioning convention before TEXT-06: new models replace the old file at the same filename. The filename is the contract, not a version indicator.
- After copying a new `.tflite` to `canaryapp/assets/models/`, run `npx expo start --clear` to force Metro cache invalidation before testing. Document this as a required step in the model replacement procedure.
- Update `MODEL_HASHES` in `ModelLoaderService.ts` with the real SHA-256 hash of the new model file using `expo-crypto` (currently a placeholder — the placeholder check is documented in `CONCERNS.md`). This turns the no-op integrity check into a regression detector.

**Phase relevance:** TEXT-06.

### Moderate Pitfall 4.4 — Vocabulary File Mismatch When Architecture Changes

**What goes wrong:** The current `TextTokenizer.ts` loads `vocab.txt` and implements WordPiece tokenization from scratch. If the architecture benchmark selects a model that uses BPE tokenization (RoBERTa, GPT-2-based) or SentencePiece (ALBERT), the existing `TextTokenizer.ts` will produce wrong token IDs that are structurally valid (correct dtype, correct shape) but semantically garbage against the new model's embedding matrix. The model will run and return a confidence score that is meaningless noise.

**Prevention:**
- Constrain the architecture benchmark (TEXT-02) to WordPiece models only: BERT-base, DistilBERT, MobileBERT, TinyBERT. These share the same `vocab.txt` and tokenization algorithm.
- If a non-WordPiece model is selected, write a new tokenizer service file and a cross-platform validation test before any model is loaded in the app. Do not modify `TextTokenizer.ts` in-place — the existing logic is the known-working baseline for the current pipeline structure.
- The 231 KB `vocab.txt` load-on-startup performance issue (documented in `CONCERNS.md`) should be addressed in TEXT-06 regardless of architecture: pre-compile to a binary `Map` and lazy-load on first scan.

**Phase relevance:** TEXT-02 and TEXT-06.

### Moderate Pitfall 4.5 — Debug Mode Logic Left in Production Path Masks Real Errors

**What goes wrong:** The current codebase has 207 `console.log` statements throughout the on-device pipeline, a visible debug panel in the home screen, and the `classify()` function throws instead of returning a degraded result. When the new model is integrated and something goes wrong (shape mismatch, wrong dtype, wrong output index), these conditions produce JavaScript exceptions that the current error handling may swallow or misattribute. The debug panel in `index.tsx` shows raw scores, which a developer might interpret as "working" even when the scores are garbage from a misconfigured input.

**Prevention:**
- Before deploying the new model (TEXT-06), remove or gate the debug panel behind `__DEV__`. This is already noted in `CONCERNS.md` — do it as a prerequisite to TEXT-06, not after.
- Replace the throw in `classify()` with a structured error return (e.g., `{ error: 'inference_failed', reason: string }`). This allows `OnDeviceScamAnalyzer.ts` to detect and log failures without crashing the analysis path.
- After integration, run the pipeline with 5 known-good and 5 known-scam texts and verify scores are in a plausible range (not all 0.5, not all 1.0) before removing any logging.

**Phase relevance:** TEXT-06.

---

### Minor Pitfall 4.6 — App Bundle Size Regression From Larger Quantized Model

**What goes wrong:** The current 26.7 MB `mobilebert_scam_intent.tflite` already exceeds the 50 MB model target when combined with the non-existent visual classifier. A distilled + quantized MobileBERT model should be smaller (target ~10–15 MB). But if the architecture benchmark selects DistilBERT without quantization as an intermediate checkpoint and that file is committed to assets, the bundle grows. The Metro bundler bundles all files in `canaryapp/assets/models/` — old model files are not excluded automatically.

**Prevention:**
- Set a hard size gate: no `.tflite` file larger than 20 MB is committed to `canaryapp/assets/models/`. Enforce this with a pre-commit check or CI step.
- The old `mobilebert_scam_intent.tflite` (26.7 MB) must be deleted — not replaced by adding a new file alongside it.

**Phase relevance:** TEXT-06.
